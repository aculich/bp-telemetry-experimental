## ADR NNNN – Short, Decision-Oriented Title

### Status

- **Status**: Proposed | Accepted | Superseded | Deprecated  
- **Date**: YYYY-MM-DD  
- **Supersedes**: (optional) ADR-XXXX  
- **Superseded by**: (optional) ADR-YYYY  

### Context

> Briefly describe the problem, background, and constraints that led to this decision.  
> Focus on information that will still matter to future readers (e.g., local-only requirement, performance goals, compatibility constraints, ecosystem considerations).

### Decision

> State the decision clearly and concretely.  
> Example: “We will use SQLite as the primary datastore for telemetry, with DuckDB as an optional analytical sink behind a feature flag.”

### Rationale

> Explain *why* this option was chosen over alternatives.  
> Call out tradeoffs, rejected options, and any important supporting data (benchmarks, ecosystem maturity, operational simplicity, privacy, etc.).

### Consequences

**Positive:**

- Bullet the main benefits and what becomes easier or clearer because of this decision.

**Negative / Tradeoffs:**

- Bullet the known downsides, risks, or things we’re explicitly not solving yet.

### Follow-ups

> List any follow-up tasks or related ADRs to write.  
> Example: “Add an ADR for where embeddings and vector search live” or “Document migration path from v1 to v2 of this decision.”
